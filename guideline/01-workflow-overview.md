# 01. 전체 워크플로 개요 (Workflow Overview)

> **대상 독자:** 고등학생도 이해 가능하도록 서비스의 전체 구조와 설계를 설명합니다.
> **프로젝트명:** 메이대구 (Maydaegu) — 꽃집 주문·배송 관리 시스템

---

## 1. 이 서비스는 뭘 하는 건가요?

**메이대구**는 대구 지역 꽃집에서 **주문이 들어왔을 때 → 꽃을 만들고 → 배송까지 완료**하는 전 과정을 한 화면에서 관리하는 사내(社內) 업무 시스템입니다.

쉽게 말하면 "꽃집 내부 직원용 카카오워크 + 주문 관리 시트" 같은 도구예요. 고객이 쓰는 쇼핑몰이 아닙니다.

### 비유로 이해하기

피자 가게를 생각해봅시다:

| 피자 가게 | 메이대구 |
|-----------|----------|
| 홀에서 주문 받는 사람 | **2층 수주 담당** (`floor2`) |
| 주방에서 피자 만드는 사람 | **1층 제작 담당** (`floor1`) |
| 피자 배달하는 사람 | **배송 기사** (`driver`) |
| 사장님 | **관리자** (`admin`) |

모든 사람이 각자 자기 역할의 화면만 보면서, 서로 같은 주문 데이터를 공유합니다.

---

## 2. 4개의 사용자 역할 (Role)

로그인하면 본인의 역할에 맞는 화면만 보입니다.

### 🟣 2층 수주 (`floor2`)
- 전화·방문으로 주문을 접수
- **신규 접수** 화면에서 체인명, 상품, 배송지, 받는분, 리본 문구를 입력
- **내 주문** 화면에서 본인이 등록한 주문만 확인

### 🔵 1층 제작 (`floor1`)
- 들어온 주문을 꽃으로 제작
- **전체 주문** 화면에서 모든 주문을 확인
- 주문 상태를 바꾸고, 제작 완료 후 **배송 기사 배정**

### 🟠 배송 기사 (`driver`)
- 본인에게 배정된 주문만 확인
- 배송 완료 시 사진 첨부
- **앞으로는 모바일 앱**(Flutter)을 주로 사용할 예정

### 🟢 관리자 (`admin`)
- 1층의 모든 기능 + **상품 관리 / 기사 관리 / 통계·정산**
- 우측 **배송기사 현황 패널**에서 실시간 기사 상태를 본다

---

## 3. 주문 한 건의 일생 (Order Lifecycle)

주문 데이터에는 `status`라는 숫자 필드가 있고, 이 숫자가 주문의 단계를 나타냅니다.

```
0,1,2 = 주문접수   (2층이 등록 → 1층이 확인/제작 중)
   3  = 배송중     (기사 배정 완료 + 출발)
   4  = 배송완료   (사진 업로드 완료)
 5,6  = 주문취소
```

### 흐름 다이어그램

```
[2층]                [1층]                [기사]
 │                    │                    │
 ├─ 신규접수 ───────► │                    │
 │   status=0         │                    │
 │                    ├─ 기사 배정 ──────► │
 │                    │   status=1~2       │
 │                    │                    ├─ 배송 출발
 │                    │                    │   status=3
 │                    │                    ├─ 사진 업로드
 │                    │                    │   status=4
 │                    │                    │   (배송완료)
```

## 4. 시스템 전체 구성도 (Architecture)

```
┌───────────────────────────────────────────────────────────┐
│                      사용자 (브라우저 / 앱)                    │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │  2층 / 1층     │    │   관리자       │    │  기사 (Flutter)│  │
│  │  (웹)        │    │   (웹)        │    │  (모바일 앱)    │  │
│  └──────┬───────┘    └──────┬───────┘    └──────┬───────┘  │
└─────────┼───────────────────┼───────────────────┼──────────┘
          │                   │                   │
          └───────────────────┼───────────────────┘
                              ▼
          ┌─────────────────────────────────────────┐
          │         Firebase (클라우드)               │
          │  ┌─────────────────┐  ┌──────────────┐   │
          │  │ Realtime DB     │  │  Storage     │   │
          │  │ (주문/채팅 등)    │  │ (사진 파일)    │   │
          │  └─────────────────┘  └──────────────┘   │
          │  ┌─────────────────┐                    │
          │  │ Auth (로그인)    │                    │
          │  └─────────────────┘                    │
          └─────────────────────────────────────────┘
```

### 핵심 원칙

1. **프론트엔드 ↔ 백엔드 분리:** 화면(웹/앱)은 보여주는 역할만, 데이터는 전부 Firebase에 저장
2. **실시간 동기화:** 한 사람이 주문을 바꾸면 다른 모든 사람 화면에도 즉시 반영 (Firebase Realtime Database)
3. **역할 기반 접근:** 로그인한 사람의 `role`에 따라 보이는 메뉴와 가능한 동작이 다름

## 5. 현재 상태 vs 앞으로의 계획

### 현재 (v1 — 프로토타입)
- **프론트엔드:** Vanilla JavaScript + HTML + CSS (프레임워크 없이 순수 코드)
- **데이터 저장:** 대부분 `localStorage` (브라우저 안에만 저장)
- **채팅만 예외:** Firebase Realtime Database로 실제 실시간 동작
- **앱:** 없음 (기사도 웹 화면으로 접속)

### 다음 단계 (v2 — 본격 개발)
| 영역 | 변경 내용 |
|------|-----------|
| 프론트엔드 | Vanilla JS → **React**로 재작성 (클론 코딩) |
| 백엔드 | localStorage → **Firebase Realtime DB + Storage + Auth** 전면 전환 |
| 앱 | **Flutter**로 기사 전용 앱 신규 개발 |
| 위치 추적 | 기사 앱에서 GPS 수집 → 대구 달서구 장기동 666-1 반경 100m 도착 시 자동 "배송대기" 전환 |

## 8. 용어 정리

| 용어 | 뜻 |
|------|-----|
| **체인명** (chainName) | 주문을 중개한 꽃집 브랜드/지점 이름 |
| **리본 문구** (ribbonText) | 화환에 붙는 축하/조의 문구 |
| **받는분** (recipientName) | 꽃을 실제로 받는 사람 |
| **배차** | 주문에 배송 기사를 배정하는 것 |
| **복귀중** | 기사가 모든 배송을 끝내고 회사로 돌아오는 중 |
| **배송대기** | 기사가 회사에 있고 배차를 받을 준비가 된 상태 |
