# 01. 프로젝트 전체 개요 및 비전

> **대상 독자:** 고등학생도 이해할 수 있도록 쉽게 설명합니다.  
> **목표:** 메이대구 시스템이 뭘 하는 건지, 왜 Flutter로 전환하는지, 전체 그림을 한눈에 파악

---

## 1. 이 서비스는 뭘 하는 건가요?

**메이대구(Maydaegu)**는 대구 지역 꽃집에서 **주문 접수 -> 꽃 제작 -> 배송 완료**까지 전 과정을 한 화면에서 관리하는 **사내(社內) 업무 시스템**입니다.

고객이 쓰는 쇼핑몰이 아닙니다. 꽃집 내부 직원들이 서로 소통하고, 주문 상태를 실시간으로 공유하기 위한 도구입니다.

쉽게 말하면 **"꽃집 내부 직원용 카카오워크 + 주문 관리 시트"** 같은 것입니다.

---

## 2. 비유로 이해하기 -- 피자 가게

피자 가게를 떠올려 보세요:

| 피자 가게 | 메이대구 |
|-----------|----------|
| 카운터에서 주문 받는 직원 | **2층 수주 담당** (`floor2`) |
| 주방에서 피자 만드는 직원 | **1층 제작 담당** (`floor1`) |
| 피자 배달하는 라이더 | **배송 기사** (`driver`) |
| 사장님 | **관리자** (`admin`) |

- 손님이 전화로 피자를 주문하면 → 카운터 직원이 주문을 시스템에 입력하고
- 주방 직원이 피자를 만들고 → 다 만들면 라이더를 배정하고
- 라이더가 배달을 출발하고 → 도착하면 사진 찍어서 완료 처리

메이대구도 정확히 이 흐름입니다. "피자" 대신 "꽃다발/화환"이 만들어지고 배달됩니다.

## 3. 4개의 사용자 역할 (Role) 상세 설명

로그인하면 본인의 역할에 맞는 화면만 보입니다.

### 2층 수주 (`floor2`)

**한 줄 요약:** 전화/방문으로 들어온 주문을 시스템에 등록하는 사람

- **신규 접수** 화면에서 체인명, 상품, 배송지, 받는분, 리본 문구 등을 입력
- **내 주문** 화면에서 본인이 등록한 주문만 확인
- 주문 상태를 변경하거나 기사를 배정할 권한은 없음
- 실시간 채팅으로 1층/기사와 소통 가능

### 1층 제작 (`floor1`)

**한 줄 요약:** 접수된 주문을 보고 꽃을 제작하고, 배송 기사를 배정하는 사람

- **전체 주문** 화면에서 모든 주문을 확인
- 주문 상태를 변경할 수 있음 (접수 확인, 제작 중 등)
- 제작 완료 후 **배송 기사 배정(배차)**
- 매장 사진 촬영 후 업로드

### 배송 기사 (`driver`)

**한 줄 요약:** 배정받은 주문의 꽃을 배송하고, 완료 사진을 찍는 사람

- 본인에게 배정된 주문만 볼 수 있음
- **배송 출발** 버튼으로 출발 기록
- 도착 후 **배송 완료 사진** 촬영 및 업로드
- GPS로 위치가 자동 공유됨 (회사 100m 반경 진입 시 자동 "배송대기" 전환)
- Flutter 전환 후에는 **모바일 앱**을 주로 사용

### 관리자 (`admin`)

**한 줄 요약:** 모든 것을 볼 수 있고, 시스템 설정을 관리하는 사장님

- 1층의 모든 기능 포함 (전체 주문 조회, 기사 배정)
- **상품 관리** (카테고리, 상품 추가/수정/삭제)
- **기사 관리** (기사 계정, 연락처, 활성화/비활성화)
- **통계/정산** (매출, 배송 건수, 기사별 실적)
- 우측 **배송기사 현황 패널**에서 실시간 기사 상태 모니터링

## 4. 주문 한 건의 일생 (Order Lifecycle)

주문 데이터에는 `status`라는 숫자 필드가 있고, 이 숫자가 주문의 현재 단계를 나타냅니다.

### 상태 코드표

| 상태 코드 | 의미 | 그룹 |
|:---------:|------|------|
| `0` | 신규 접수 (2층이 방금 등록) | 주문접수 |
| `1` | 접수 확인 (1층이 확인함) | 주문접수 |
| `2` | 제작 중 / 배차 완료 | 주문접수 |
| `3` | 배송중 (기사가 출발) | 배송중 |
| `4` | 배송완료 (사진 업로드 완료) | 배송완료 |
| `5` | 주문취소 A | 주문취소 |
| `6` | 주문취소 B | 주문취소 |

### 흐름 다이어그램

```
[2층 수주]              [1층 제작]              [배송 기사]
   │                      │                      │
   ├── 주문 등록 ─────────►│                      │
   │   status = 0          │                      │
   │                      ├── 접수 확인            │
   │                      │   status = 1           │
   │                      │                      │
   │                      ├── 제작 중 + 기사 배정 ─►│
   │                      │   status = 2           │
   │                      │                      │
   │                      │                      ├── 배송 출발
   │                      │                      │   status = 3
   │                      │                      │
   │                      │                      ├── 배송 도착 + 사진 업로드
   │                      │                      │   status = 4  (배송완료)
   │                      │                      │
   ├──── 취소 가능 ────────┤                      │
   │   status = 5 or 6     │                      │
```

**핵심 규칙:**
- 상태 코드는 **절대 변경하지 않는 규약**입니다 (웹과 앱 모두 동일)
- 0,1,2 = 주문접수 / 3 = 배송중 / 4 = 배송완료 / 5,6 = 주문취소

## 6. Flutter 통합 비전 -- 왜 전환하는가?

### 현재 문제점

현재 시스템은 **Vanilla JavaScript + HTML + CSS**로 만들어진 웹 프로토타입입니다.

- 데이터 대부분이 `localStorage`(브라우저 안에만 저장)에 있어서 다른 기기에서 볼 수 없음
- 배송 기사도 PC 브라우저로 접속해야 함 (이동 중 사용 불가)
- 카메라, GPS, 푸시 알림 같은 모바일 기능을 쓸 수 없음
- 웹과 앱을 따로 만들면 코드를 두 벌 관리해야 함

### Flutter로 전환하면?

**Flutter**는 Google이 만든 프레임워크로, **하나의 코드베이스**로 웹/Android/iOS 앱을 동시에 만들 수 있습니다.

| 문제 | Flutter 해결책 |
|------|---------------|
| 기사 전용 앱이 없음 | 같은 코드로 Android + iOS 앱 자동 빌드 |
| 웹/앱 따로 개발 필요 | **단일 코드베이스**로 Web + Android + iOS 동시 배포 |
| 카메라, GPS 사용 불가 | Flutter 플러그인으로 네이티브 기능 직접 접근 |
| 개발 속도 느림 | **Hot Reload**로 코드 수정 즉시 반영 (저장하면 바로 화면 변경) |
| 플랫폼별 UI 따로 작성 | Flutter가 플랫폼에 맞게 자동 적응 (Material Design + Cupertino) |

### 구체적 이점

1. **단일 코드베이스:** 관리자/사무직은 `flutter run -d chrome`으로 웹 대시보드를 쓰고, 기사는 같은 앱을 안드로이드/아이폰에 설치해서 사용. 코드 하나로 3개 플랫폼.
2. **기사 앱 별도 개발 불필요:** 기존 계획에서는 React(웹) + Flutter(기사앱)을 따로 만들려 했지만, Flutter 하나로 통합하면 코드 중복 없음.
3. **네이티브 기능 활용:** 카메라(배송 완료 사진), GPS(위치 추적), 푸시 알림(배차 알림)을 Flutter 플러그인으로 바로 사용.
4. **Hot Reload:** 코드를 수정하고 저장하면 1초 안에 화면에 반영. 개발 속도가 크게 향상.

## 7. v1(현재) vs v2(Flutter 통합) 비교표

| 항목 | v1 (현재 프로토타입) | v2 (Flutter 통합) |
|------|---------------------|-------------------|
| **프론트엔드** | Vanilla JS + HTML + CSS | Flutter (Dart) |
| **플랫폼** | 웹 브라우저만 | Web + Android + iOS |
| **데이터 저장** | localStorage (브라우저 내부) | Firebase Realtime Database (클라우드) |
| **채팅** | Firebase (이미 구현) | Firebase (유지) |
| **로그인** | 평문 비밀번호 (`'1234'`) | Firebase Authentication |
| **사진 저장** | 없음 / 제한적 | Firebase Cloud Storage |
| **기사 앱** | 없음 (웹으로 접속) | Flutter 네이티브 앱 (Android + iOS) |
| **GPS 추적** | 없음 | `geolocator` 플러그인으로 실시간 위치 |
| **푸시 알림** | 없음 | Firebase Cloud Messaging (FCM) |
| **오프라인 지원** | 없음 | Firebase 오프라인 캐시 |
| **개발 도구** | 텍스트 에디터 + 브라우저 | VS Code/Android Studio + Hot Reload |
| **배포** | GitHub Pages / 수동 | Firebase Hosting(웹) + Play Store/App Store(앱) |
| **코드베이스** | 1개 (웹 전용) | 1개 (Web + Android + iOS 통합) |

## 9. 용어 정리

| 용어 | 영문 키 | 뜻 |
|------|---------|-----|
| **체인명** | `chainName` | 주문을 중개한 꽃집 브랜드/지점 이름 (예: "행복꽃집 동성로점") |
| **리본 문구** | `ribbonText` | 화환이나 꽃바구니에 붙는 축하/조의 문구 (예: "개업을 축하합니다") |
| **받는분** | `recipientName` | 꽃을 실제로 받는 사람 이름 |
| **배차** | assign driver | 주문에 배송 기사를 배정하는 행위 |
| **배송대기** | `waiting` | 기사가 회사에 있고, 새 배차를 받을 준비가 된 상태 |
| **배차완료** | `assigned` | 기사에게 주문이 배정되었으나 아직 출발하지 않은 상태 |
| **배송중** | `delivering` | 기사가 배송을 위해 출발한 상태 |
| **복귀중** | `returning` | 기사가 모든 배송을 마치고 회사로 돌아오는 중인 상태 |
| **즉시 배송** | `isImmediate` | 일반 예약이 아닌 즉시 배송 주문 |
| **매장 사진** | `storePhotoUrl` | 출발 전 매장에서 완성된 꽃 사진 |
| **배송 완료 사진** | `deliveryPhotoUrl` | 배송지에 도착하여 전달 후 찍은 사진 |
| **resetTs** | `resetTs` | 기사 타임라인 초기화 기준 시각 (회사 100m 진입 시 갱신) |
| **Firebase** | - | Google 클라우드 서비스. 실시간 DB, 파일 저장, 로그인 등 제공 |
| **Hot Reload** | - | Flutter에서 코드 수정 시 앱을 재시작하지 않고 즉시 반영하는 기능 |
| **Dart** | - | Flutter 앱을 작성하는 프로그래밍 언어 |
| **Riverpod** | - | Flutter의 상태 관리 라이브러리. 데이터 흐름을 깔끔하게 관리 |
